FiveStarz Project Summary
FiveStarz is a Next.js-based SaaS application for online entrepreneurs seeking authentic reviews through ethical peer exchange.
Core Concept
A members-only club where founders and solopreneurs exchange honest feedback on each other's products/services, then mutually decide whether to post that feedback as public reviews on platforms like Google, Yelp, Amazon, etc.
Key Features
	•	Smart Matching System: Connects members based on "degrees of separation" (1°/2°/3°) to ensure genuine connections
	•	Multi-Asset Management: Users can create multiple "assets" (products/services) to collect reviews
	•	Multi-Channel Reviews: Each asset supports multiple review platforms (Google Business, Yelp, TripAdvisor, Amazon, Shopify)
	•	Feedback Types: Text, star ratings, video testimonials, and "Advisory Skills" feedback
	•	3-Tier Pricing: Sprout (Free), Bloom ($29/mo - most popular), Flourish ($79/mo)
Current State
	•	MVP/Prototype phase with mock data (no backend/auth yet)
	•	4 main pages: Landing page, How It Works, Dashboard (with matches/assets/history), Asset Setup Flow
	•	Beta signup form integrated (configured for Resend email to spasta+fivestarz@gmail.com)
	•	Built with React 19, Next.js 16, warm color palette (teal/orange/gold/brown/cream)
The app emphasizes ethical, consent-based review generation rather than pestering customers.

Recommended Core Data Model
1) Users
A user is a member of the club and can own multiple assets.
Key fields:
	•	id
	•	email
	•	name
	•	avatar_url
	•	bio
	•	location
	•	plan (sprout, bloom, flourish)
	•	degrees_of_separation (1, 2, 3)
	•	allow_semi_duplicate_matches
	•	allow_semi_duplicate_with_free
	•	feedback_quality_rating
	•	created_at
	•	updated_at
Likely additional usage fields:
	•	monthly_auto_matches_used
	•	monthly_browse_matches_used
	•	monthly_match_limit
	•	monthly_browse_limit
In production, I’d probably keep usage in a separate monthly usage table instead of storing counters directly on users.

2) Assets
An asset is the thing being reviewed: service, SaaS, podcast, advisory skill, client asset, etc.
Key fields:
	•	id
	•	owner_user_id → users.id
	•	name
	•	url
	•	type
	•	description
	•	is_client_asset
	•	client_name (nullable)
	•	status (draft, pending_verification, active, archived)
	•	require_star_rating
	•	require_star_plus_one_other
	•	created_at
	•	updated_at
Supporting asset-related tables you’ll likely need:
	◦	asset_channels
	◦	id
	◦	asset_id
	◦	channel_name (Google Business Profile, Yelp, Trustpilot, etc.)
	◦	channel_url
	◦	asset_feedback_formats
	◦	asset_id
	◦	format (stars, text, structured, video)
	◦	asset_screenshots
	◦	id
	◦	asset_id
	◦	file_url
	◦	sort_order
The mock UI clearly treats channels and feedback types as multi-value fields, so these should not be hardcoded into one string column.

3) Matches
A match is the exchange between two users and usually involves one asset from each side.
This is the most important model because it drives:
	•	duplicate prevention
	•	semi-duplicate logic
	•	blocked channels
	•	status pipeline
	•	graph queries for degrees of separation
Key fields:
	•	id
	•	user_a_id
	•	user_b_id
	•	asset_a_id
	•	asset_b_id
	•	source (auto, browse, queued)
	•	status (matched, accepted, feedback_pending, awaiting_post, posted, completed, cancelled, queued_next_month)
	•	feedback_due_at
	•	is_semi_duplicate
	•	previous_match_id (nullable)
	•	separation_degree_used
	•	created_at
	•	updated_at
You also need match-level channel restrictions for semi-duplicates:
	◦	match_blocked_channels
	◦	id
	◦	match_id
	◦	asset_id
	◦	channel_name
That lets you enforce:
	•	“these two users matched before”
	•	“they may only match again if different channels are available”
	•	“previous channels are blocked on both sides”

4) Reviews
This is the one model I would not keep as a single simple table.
The product has two different concepts:
A. Internal feedback
This is submitted inside FiveStarz.
Fields:
	•	id
	•	match_id
	•	reviewer_user_id
	•	reviewee_user_id
	•	asset_id
	•	star_rating (nullable)
	•	written_feedback (nullable)
	•	structured_feedback_json (nullable)
	•	media_url (nullable)
	•	submitted_at
B. Public review posting
This happens only if the asset owner requests it and the reviewer agrees.
Fields:
	•	id
	•	feedback_id → internal feedback
	•	requested_channel
	•	request_status (pending, accepted, declined, posted)
	•	posted_url (nullable)
	•	posted_at (nullable)
	•	confirmed_at (nullable)
So conceptually:
one internal feedback record may optionally become one public posted review

	◦	one from B → A
	•	Internal feedback can optionally produce a public review post
